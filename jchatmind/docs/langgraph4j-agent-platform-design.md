# 基于 LangChain4j + LangGraph4j + LangSmith 的多智能体执行平台设计文档

## 0. 项目名称与定位

### 0.1 项目名称

建议项目名称为：

**FlowCopilot AI**

可选中文名：

**流协同多智能体执行平台**

可选副标题：

**面向复杂任务执行的多智能体工作流平台**

### 0.2 项目定位

本项目不是普通聊天机器人，也不是单纯的知识库问答系统，而是一个：

**以自然语言为入口、以任务交付为目标、以多智能体协作为执行机制、以工作流图为运行骨架、以知识增强和可观测性为支撑的多智能体执行平台。**

它的核心定位是：

- 用户通过自然语言描述任务
- 系统将任务转化为状态化工作流
- 多个角色型 Agent 协同完成任务
- 系统输出结构化交付结果
- 整个过程可追踪、可回放、可人工审核

### 0.3 项目一句话描述

FlowCopilot AI 是一个能够将“聊天输入”转化为“任务执行流程”的多智能体平台，面向复杂任务的理解、拆解、检索、执行、复核和交付全过程。

## 1. 背景

当前项目已经具备一个典型 Agent 原型系统的基础能力：

- 支持聊天会话管理
- 支持 Agent 配置与模型切换
- 支持工具调用
- 支持知识库上传与基础 RAG
- 支持 SSE 实时返回 Agent 输出

从实现上看，现有系统的核心运行链路主要集中在以下几个模块：

- `Agent` 运行时核心：[JChatMind.java](/Users/anastasio/Programming/JChatMind/jchatmind/src/main/java/aliang/flowcopilot/agent/JChatMind.java)
- `Agent` 装配工厂：[JChatMindFactory.java](/Users/anastasio/Programming/JChatMind/jchatmind/src/main/java/aliang/flowcopilot/agent/JChatMindFactory.java)
- 消息触发事件：[ChatEventListener.java](/Users/anastasio/Programming/JChatMind/jchatmind/src/main/java/aliang/flowcopilot/event/listener/ChatEventListener.java)
- RAG 检索实现：[RagServiceImpl.java](/Users/anastasio/Programming/JChatMind/jchatmind/src/main/java/aliang/flowcopilot/service/impl/RagServiceImpl.java)

但当前架构仍然有明显边界：

- 更像“单 Agent 对话系统”，而不是“多智能体协作系统”
- 更像“消息驱动输出”，而不是“任务驱动交付”
- 更像“单轮工具调用循环”，而不是“工作流图编排”
- 更像“基础 RAG + Chat”，而不是“流程化知识增强系统”
- 可观测性偏弱，难以系统化做调试、评测和回放

与此同时，市场和竞赛类题目对 Agent 产品的期待已经从“聊天机器人”升级为：

- 能拆任务
- 能组织协作
- 能调用工具
- 能走流程
- 能和人协同审批
- 能输出结构化交付物
- 能解释执行过程

因此，本次改造的目标不是简单替换模型 SDK，而是完成一次**能力层级升级**：

- 从 `Single-Agent Loop` 升级到 `Multi-Agent Graph`
- 从 `Chat System` 升级到 `Execution Platform`
- 从 `Prompt + Tool` 升级到 `State + Workflow + Observability`

技术栈选择原因如下：

- `LangChain4j`
  适合 Java 技术栈下的模型接入、工具调用、结构化输出与 RAG 封装。
- `LangGraph4j`
  适合将复杂任务建模为状态图与工作流，支持节点编排、并行执行、条件跳转、可恢复执行。
- `LangSmith`
  适合承接全链路 Trace、效果观测、回放调试与评测闭环。

最终目标是将当前项目升级为：

**一个面向复杂任务协同的多智能体执行平台。**

## 1.1 真实场景

虽然项目不绑定某个固定赛题业务，但它具备明确的真实应用场景。

### 场景一：研究分析场景

用户希望快速完成调研、分析和总结类任务，例如：

- 对某个主题进行资料整理
- 汇总多个来源内容形成分析报告
- 提取核心结论与建议

传统做法需要人工反复查资料、整理信息、归纳结构、撰写总结，过程耗时且重复劳动多。  
本项目可以通过多智能体协同完成：

- 任务理解
- 资料检索
- 内容整合
- 结果复核
- 最终报告输出

### 场景二：办公协同场景

用户在日常办公中经常遇到以下任务：

- 根据会议内容生成纪要
- 提取行动项与负责人
- 生成汇报提纲
- 整理任务进度摘要

这类任务通常需要对话理解、知识检索、结构化提取和内容生成共同参与，适合用工作流和多 Agent 协同完成。

### 场景三：项目推进场景

用户在项目过程中常常需要：

- 拆解需求
- 生成执行计划
- 梳理风险
- 汇总当前状态
- 输出结构化任务单

这些任务并不是简单问答，而是一个持续推进的执行链。本项目适合作为这类任务的执行骨架。

### 场景四：知识驱动执行场景

很多团队或组织拥有大量资料：

- 规范文档
- 历史案例
- 模板
- 研究材料
- 方案沉淀

但这些知识通常停留在“被动存储”层面。  
本项目希望解决的问题是：

**让知识进入执行流程，真正参与任务决策与交付。**

## 1.2 项目要解决的需求

本项目要解决的不是“聊天问答”的单点需求，而是复杂任务执行过程中的系统性需求。

### 需求一：复杂任务自动化执行需求

用户面对复杂任务时，通常需要：

- 先理解目标
- 再查找资料
- 再整理信息
- 再生成结果
- 再人工复核

这一流程高度依赖人工串联，效率低且容易遗漏。本项目希望将这些步骤流程化、状态化和智能化。

### 需求二：多角色协同需求

复杂任务往往不是一个“万能助手”就能稳定完成的。  
系统需要具备：

- 拆解任务的角色
- 检索知识的角色
- 执行生产的角色
- 复核把关的角色
- 汇总交付的角色

换句话说，系统要从“单 Agent 对话”升级为“多角色协同执行”。

### 需求三：知识增强执行需求

组织中的文档、历史方案、案例、规则和模板如果不能参与执行，就无法真正提升效率。  
因此，项目需要：

- 接入知识库
- 将检索作为流程节点
- 支持知识引用和结果溯源

### 需求四：人机协同需求

并非所有任务都适合完全自动完成。  
系统需要在关键节点支持：

- 人工确认
- 驳回重试
- 补充信息
- 中断恢复

### 需求五：可观测和可优化需求

如果系统无法解释执行过程，就无法稳定迭代。  
本项目需要：

- 记录模型调用
- 记录工具调用
- 记录检索过程
- 记录节点输入输出
- 支持回放和评测

## 1.3 需求确定

结合项目现状与未来演进方向，建议将本项目的需求正式确定为：

### 项目目标

构建一个基于 LangChain4j、LangGraph4j 和 LangSmith 的多智能体执行平台，用于解决复杂任务在“理解、拆解、检索、执行、复核、审批、交付”过程中的自动化协同问题。

### 项目面向场景

- 研究分析
- 办公协同
- 项目推进
- 知识驱动执行

### 核心需求定义

1. 用户能够通过自然语言提交复杂任务。
2. 系统能够将任务转化为状态化工作流。
3. 系统能够通过多个角色 Agent 协同完成任务。
4. 系统能够在执行过程中调用知识库和外部工具。
5. 系统能够在关键节点支持人工审核与继续执行。
6. 系统能够输出结构化交付结果，而不仅仅是聊天回复。
7. 系统能够以可视化方式展示执行流程、节点状态和中间结果。
8. 系统能够记录完整执行链路，用于调试、评测和回放。

## 2. 能力提炼

虽然业务背景可以自由定义，但从当前行业趋势与高质量 Agent 系统实践中，可以稳定提炼出以下八类核心能力。

### 2.1 多角色协作能力

系统不再由单个 Agent 独立完成全部任务，而是引入多个有职责边界的角色型 Agent，例如：

- Planner Agent：负责理解目标并生成计划
- Retriever Agent：负责检索知识和补充上下文
- Executor Agent：负责执行主要生产任务
- Reviewer Agent：负责复核与质量控制
- Reporter Agent：负责汇总结果并生成最终交付物

这类能力的价值在于：

- 降低单个 Prompt 负担
- 提高复杂任务的可控性
- 让系统具备“团队协同”的产品感

### 2.2 工作流编排能力

复杂任务不是一轮对话即可完成，而是多个节点逐步推进。工作流能力要求系统能够：

- 将任务拆成多个节点
- 为节点定义输入、输出、状态与跳转条件
- 允许串行和并行执行
- 支持失败重试与人工打断

这要求平台从“对话式循环”升级为“状态驱动的图执行模型”。

### 2.3 Human-in-the-loop 能力

真实业务里并不是所有节点都适合完全自动执行，关键节点需要人工确认，例如：

- 是否接受 Agent 制定的计划
- 是否通过生成结果
- 是否确认进入下一阶段
- 是否补充信息后重新执行

这种人机协同能力会显著提升系统的可信度、可解释性和落地性。

### 2.4 任务交付能力

系统的输出不能只是一段聊天文本，而应当是可被业务使用的交付物，例如：

- 报告
- 文档
- 表格
- 任务单
- 清单
- 审批意见
- 分析摘要

换句话说，系统要从“回答问题”升级到“推进事情完成”。

### 2.5 状态管理能力

要支撑复杂协作和图执行，必须显式管理状态，而不是把所有信息都塞进聊天历史中。系统需要维护：

- 流程实例状态
- 当前节点状态
- 任务状态
- 审批状态
- 各 Agent 的中间结果
- 引用的知识与产物

状态是图执行系统的骨架，也是后续恢复执行和回放分析的基础。

### 2.6 工具编排能力

Agent 的价值离不开工具调用。平台需要支持统一工具能力中心，至少包括：

- 知识库检索工具
- 数据库查询工具
- 文件读写工具
- 文档生成工具
- 第三方接口工具
- 通知与消息工具

关键不是“支持工具调用”本身，而是“支持按流程和角色进行工具编排”。

### 2.7 知识增强能力

RAG 不应该只是聊天功能的附属能力，而应变成流程节点的标准输入来源。知识增强体系需要支持：

- 知识库分域管理
- 文档切分与索引
- 检索与重排
- 检索结果追踪
- 输出结果引用来源

这使系统具备组织记忆和业务上下文理解能力。

### 2.8 可观测与评测能力

Agent 系统如果没有观测能力，后期几乎无法稳定优化。成熟系统必须支持：

- 每次模型调用 Trace
- 工具调用 Trace
- 检索过程 Trace
- 节点级输入输出记录
- 流程回放
- 评测集验证

这正是 LangSmith 需要重点发挥的价值。

## 3. 目标亮点

本次改造后的项目，建议对外强调以下产品亮点。

### 3.1 多智能体协作执行

平台支持多角色 Agent 协同执行复杂任务，不再依赖单个 Agent 完成全部推理和执行。每个 Agent 有明确职责边界、工具权限和知识访问范围。

### 3.2 图式工作流执行

平台采用 LangGraph4j 将执行过程建模为状态图，支持：

- 节点化执行
- 条件分支
- 并行处理
- 中断恢复
- 人工审批插入

相比传统单 Agent Loop，可解释性与工程可控性更强。

### 3.3 知识增强型执行链路

知识检索不再局限于问答场景，而是在工作流中作为标准节点参与执行决策。平台支持从知识库中引入背景、规则、模板和历史案例，提升执行质量。

### 3.4 交付物导向的结果输出

平台输出面向业务使用的交付结果，而非仅输出自然语言回复。可以进一步扩展生成：

- 报告
- 方案草稿
- 审批建议
- 检查清单
- 任务拆解结果

### 3.5 人机协同

系统支持在关键节点插入人工操作，形成“AI 执行 + 人工把关”的混合模式，既提升效率，也降低误判风险。

### 3.6 全链路可观测

平台通过 LangSmith 对模型调用、节点执行、工具调用和知识检索进行全链路追踪，实现：

- 可调试
- 可评估
- 可回放
- 可解释

### 3.7 平台化扩展能力

新架构不绑定特定业务背景，可以在后续按需扩展为：

- 办公协同助手
- 研发流程助手
- 项目推进助手
- 研究分析助手
- 内容生产助手

这使系统具备二次包装和持续演进空间。

## 4. 总体架构

建议整体采用六层架构。

### 4.1 接入层

负责接收用户输入、外部系统事件和前端交互请求。

组成：

- Web 前端
- 聊天入口
- 审批入口
- 执行看板入口
- 第三方 IM / Webhook 接入

职责：

- 接收自然语言任务
- 展示执行状态
- 展示流程节点
- 展示产物与审批

### 4.2 编排层

负责定义与调度工作流，是系统的业务骨架。

组成：

- Workflow Definition Service
- Workflow Runtime Service
- Workflow Scheduler
- Approval Service

职责：

- 创建流程实例
- 驱动图执行
- 管理节点跳转
- 管理人工审批与恢复

### 4.3 Agent 层

负责角色化执行逻辑。

组成：

- Planner Agent
- Retriever Agent
- Executor Agent
- Reviewer Agent
- Reporter Agent

职责：

- 接收状态对象
- 执行角色职责
- 输出结构化中间结果

### 4.4 AI 能力层

负责承接大模型、Embedding、工具调用与结构化输出。

组成：

- LangChain4j Chat Model Adapter
- Tool Registry
- Embedding Service
- Retriever Service
- Structured Output Parser

职责：

- 调用模型
- 结构化解析结果
- 调用外部工具
- 完成知识检索

### 4.5 数据层

负责状态与业务数据持久化。

建议：

- `MySQL 5.7`：业务库
- `Qdrant / Milvus / PostgreSQL+pgvector`：向量库
- 对象存储：文件与产物
- Redis：缓存、临时状态、任务队列

### 4.6 可观测层

负责观测、评测和优化闭环。

组成：

- LangSmith
- 日志系统
- 指标系统
- 失败样本集
- Prompt / Workflow 调优面板

职责：

- 采集 Trace
- 分析失败链路
- 对比不同 Prompt / Graph 版本效果
- 回放执行过程

## 5. 核心模块

### 5.1 Workflow Engine

这是新系统的核心模块，负责将任务执行建模为图。

关键能力：

- 定义状态对象 `WorkflowState`
- 定义节点接口 `WorkflowNode`
- 定义条件路由器
- 管理节点执行顺序
- 支持恢复、暂停、重试

输入：

- 用户任务
- 流程定义
- 会话上下文

输出：

- 工作流实例
- 节点执行结果
- 任务推进状态

### 5.2 Agent Role Center

负责管理角色型 Agent 的运行配置。

配置内容：

- 角色名称
- 提示词
- 允许调用的工具
- 可访问知识域
- 输出格式

作用：

- 让 Agent 的角色边界清晰
- 降低系统整体 Prompt 耦合度
- 方便后续策略迭代

### 5.3 Task Center

负责统一任务视角。

功能：

- 创建任务
- 任务拆解
- 任务分派
- 状态推进
- 输出负责人、截止时间、优先级

作用：

- 让系统从“聊天消息”升级成“任务驱动”

### 5.4 Approval Center

负责人工审核与流程接管。

支持动作：

- 通过
- 驳回
- 补充信息
- 重试节点
- 终止流程

作用：

- 避免黑盒自动推进
- 增强可信度
- 满足真实业务约束

### 5.5 Knowledge Center

负责知识库、索引与检索策略。

功能：

- 文档上传
- 文档切分
- 索引构建
- 检索与重排
- 检索引用记录

能力目标：

- 让检索成为平台基础能力，而不是聊天附属功能

### 5.6 Artifact Center

负责管理执行产物。

产物类型：

- 文本报告
- 分析结果
- 审批结果
- 结构化任务清单
- 文件型产物

作用：

- 将执行过程固化为业务可用结果

### 5.7 Trace Center

负责每条执行链路的观测。

记录内容：

- 请求级 Trace
- 节点级 Trace
- Agent 输入输出
- 工具调用结果
- 检索结果
- 错误原因

作用：

- 为 Prompt 优化、Graph 优化和线上稳定性提供数据支撑

### 5.8 Operation Dashboard

负责前端执行看板。

展示内容：

- 当前流程图
- 当前节点
- 节点执行日志
- 待审批事项
- 生成产物
- 执行耗时
- 历史 Trace

这是比赛展示和项目答辩时非常重要的视觉亮点。

## 7. 演进路线

建议按“三层演进”推进。

### 第一阶段：L1 能力升级

从当前系统升级到“增强版单 Agent”。

特征：

- 保留聊天入口
- 引入 LangChain4j
- 规范 Tool 与 RAG 封装
- 引入基本 Trace

目标：

- 完成模型与工具体系重构

### 第二阶段：L2 协同执行

从增强版单 Agent 升级到“多 Agent 协同系统”。

特征：

- 多角色 Agent
- 节点化执行
- 状态对象
- 审批节点

目标：

- 完成复杂任务分工与协作闭环

### 第三阶段：L3 平台化落地

从协同执行系统升级到“平台级执行引擎”。

特征：

- 完整 Graph
- 子图复用
- 中断恢复
- 执行面板
- LangSmith 评测闭环

目标：

- 构建可扩展、可复用、可展示的平台型产品

## 8. MVP 范围

建议第一版 MVP 不求做大而全，只求体现核心亮点。

### 8.1 MVP 目标

建立一条最小可用的多智能体执行链路：

`用户输入 -> Planner -> Retriever -> Executor -> Reviewer -> Approval -> Publish`

### 8.2 MVP 必须具备的能力

- 聊天或文本入口
- 多 Agent 协作
- Graph 工作流
- 至少一个 RAG 节点
- 至少一个人工审批节点
- 至少一个可视化执行面板
- LangSmith Trace

### 8.3 MVP 输出结果

至少支持输出一种交付物，例如：

- 分析报告
- 执行建议
- 任务清单
- 结构化方案摘要

### 8.4 MVP 不做的内容

首版不建议立刻做：

- 多业务大而全覆盖
- 复杂权限体系
- 多租户
- 全量插件市场
- 复杂 BI 统计
- 完整移动端

### 8.5 MVP 成功标准

满足以下条件即可认为 MVP 成功：

- 能稳定创建流程实例
- 能驱动至少 4 个节点执行
- 能完成一次知识检索增强
- 能进入人工审批节点
- 能生成最终结果
- 能在 LangSmith 中查看完整执行链路
- 能在前端看到节点状态与执行过程

## 9. 需求分析补充

为了便于后续方案汇报、产品说明和答辩表达，补充给出更清晰的需求分析。

### 9.1 目标用户

项目的首要目标用户建议聚焦为以下几类：

- 需要频繁进行资料整理和输出的人
- 需要把复杂任务拆解后推进的人
- 需要借助知识库和模板提高效率的人
- 需要对 AI 执行过程进行可视化观察和审核的人

典型用户可以包括：

- 学生或研究者
- 办公协同用户
- 小团队项目参与者
- 内容整理与知识运营人员

### 9.2 业务痛点

目标用户普遍存在以下痛点：

- 复杂任务处理步骤多，人工串联耗时
- 查找知识与实际执行脱节
- 单一 Agent 不够稳定，结果质量波动大
- 生成结果之后缺乏复核机制
- 无法追踪系统为何做出某个决策
- 无法方便地把聊天输入转化为交付结果

### 9.3 功能需求

系统需要具备以下功能：

- 聊天式任务输入
- 任务状态维护
- 多角色 Agent 协作
- 图式工作流执行
- 知识库检索与引用
- 人工审批节点
- 结构化结果输出
- 执行过程可视化
- Trace 记录与回放

### 9.4 非功能需求

系统应满足以下非功能要求：

- 执行过程可解释
- 节点失败可恢复
- 关键结果可人工确认
- 中间结果可持久化
- 检索过程可追踪
- 输出尽量结构化
- 前端具备流程展示能力
- 后端具备扩展更多工作流模板的能力

## 10. 结语

本次改造的重点不在于“换一个 AI SDK”，而在于：

- 引入状态思维
- 引入图执行思维
- 引入多角色协同思维
- 引入可观测与评测思维

只有完成这四层升级，项目才会真正从“基础 Agent Demo”升级为“多智能体执行平台原型”。

- 代码架构与包结构重构设计
